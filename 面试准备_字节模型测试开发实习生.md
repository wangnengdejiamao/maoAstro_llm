## 一、项目详细介绍（小白向）

### 1.1 项目一句话介绍

**maoAstro LLM** 是一个面向天文学领域的AI智能分析平台，让天文学家可以用自然语言对话的方式，自动获取、分析天体数据，并生成专业分析报告。

### 1.2 为什么要做这个项目？

想象一下，一位天文学家想研究一颗恒星：
- 他需要去不同的数据库查数据（ZTF、TESS、LAMOST、Gaia...）
- 每个数据库有自己的查询方式，要手动写代码
- 拿到数据后还要自己画图、做分析
- 最后才能得出科学结论

**耗时耗力！** 我的项目就是想用AI把这个流程自动化。

### 1.4 三大核心功能详解

#### 🔹 功能一：大模型本地部署

**什么是本地部署？**

通常我们用的ChatGPT是在云端运行的，需要联网。我的项目把大模型部署在自己的电脑上：
- 用 **Ollama** 框架（一个本地运行大模型的工具）
- 支持多种模型切换：Qwen3:8b、Llama3.1、DeepSeek
- 好处：数据不泄露、响应快、无需联网

**面试时可以这样说：**
> "我使用Ollama框架实现了Qwen、Llama等模型的本地部署，让 astronomers 可以在没有网络的情况下使用AI分析私密的天文数据，保障了数据安全性。"

#### 🔹 功能二：双轨RAG系统

**什么是RAG？**

RAG = Retrieval Augmented Generation（检索增强生成）

简单说：大模型有知识截止日期，不知道最新信息。RAG就是让大模型先查资料，再回答。

**我的"双轨"是什么意思？**

| 轨道 | 名称 | 原理 | 适用场景 |
|------|------|------|----------|
| 轨道一 | 向量检索RAG | 把知识转成向量，语义相似度匹配 | 理解性问题，如"什么是白矮星" |
| 轨道二 | 关键词RAG | 倒排索引，精确关键词匹配 | 专业术语查询，如"AM CVn周期范围" |

**为什么需要双轨？**

天文领域有很多精确的专业术语，比如：
- 问"AM CVn"是什么，向量检索能答
- 问"AM CVn的周期范围"，需要精确数字（5-65分钟），关键词检索更准确

**面试时可以这样说：**
> "我设计了双轨RAG系统，结合向量检索的语义理解能力和倒排索引的精确匹配能力，解决了天文领域专业术语召回不准确的问题。比如查询'AM CVn'时，能准确召回其周期范围5-65分钟的关键信息。"

#### 🔹 功能三：自动化工作流

**什么是工作流？**

就像一个流水线：
1. **意图理解**：用户说"分析这颗恒星"→ AI理解要做什么
2. **数据查询**：自动去各个数据库拉数据
3. **数据处理**：画图、计算、分析
4. **报告生成**：生成PDF/JSON格式的分析报告

**用了什么技术？**

- **LangGraph**：把上面的流程画成"图"，每个节点是一个Agent
- **多模态VLM**：不仅处理文字，还能"看懂"天文图片
- **40+ Python模块**：封装了所有天文数据处理逻辑

**面试时可以这样说：**
> "我使用LangGraph构建了多Agent协作的工作流，实现了'意图理解→数据查询→处理→报告生成'的全流程自动化。同时接入了VLM视觉模型，可以分析天文图像。"

### 1.5 技术亮点补充

#### 🔹 模型蒸馏

**什么是模型蒸馏？**

大模型（如70B参数的Qwen）效果很好但太慢，小模型（如8B）快但笨。模型蒸馏就是让"小模型学习大模型的思维"。

**我的做法：**
- 用LangGraph生成训练数据（大模型思考的中间过程）
- 训练小模型模仿大模型的输出分布
- 用LoRA技术，只训练部分参数，节省资源

#### 🔹 幻觉检测机制

**什么是幻觉？**

大模型"一本正经地胡说八道"。比如我问"AM CVn含有什么星？"，如果模型说"中子星"，就是幻觉（正确答案是白矮星+氦星）。

**我的解决办法：**
- RAG知识库中标记"常见错误"
- 在Prompt里加警示："回答前检查是否有中子星"
- 多源数据交叉验证

#### Q：如何评估你这个系统的效果？

**回答思路（分层次）：**

```
评估维度

│   ├── 召回率：专业术语能否被正确召回
│   ├── 精确率：召回的内容是否相关
│   └── 延迟：检索耗时
│
├── 2. 生成效果评估（大模型部分）
│   ├── 人工评估：请天文专业同学打分
│   ├── 自动指标：ROUGE-L、BERTScore
│   └── 幻觉检测：事实一致性检查
│
├── 3. 端到端评估
│   ├── 任务完成率：用户请求是否被满足
│   ├── 用户满意度：实际使用反馈
│   └── 报告质量：生成报告的专业性
│
└── 4. 工程指标
    ├── 响应延迟：平均响应时间
    ├── 并发能力：同时处理多少请求
    └── 资源占用：显存、CPU使用率
```

**详细回答：**

> "我从四个维度评估系统效果：
> 
> **第一是检索效果**，我使用天文领域的专业问答对作为测试集，计算RAG的Recall@K和Precision。比如查询'AM CVn'，系统应该能召回包含'白矮星'、'氦星'、'5-65分钟周期'等关键信息的文档。
> 
> **第二是生成效果**，我采用了人工+自动结合的方式。人工方面，我请天文专业的朋友对50个样本进行打分；自动方面，使用BERTScore计算生成文本与参考文本的语义相似度。
> 
> **第三是幻觉检测**，我整理了天文领域的'常见错误清单'，在测试集中故意加入诱导性问题（如'AM CVn是否含有中子星'），检查模型是否会犯错。
> 
> **第四是工程指标**，包括平均响应时间（目标<3秒）、并发处理能力等。"

#### Q：大模型幻觉问题怎么处理？

**回答思路（分层防御）：**

```
幻觉防御体系
├── 1. 事前防御（预防）
│   ├── RAG增强：提供准确上下文
│   ├── Prompt工程：加入"不确定就说不知道"的约束
│   └── 温度调节：降低temperature减少随机性
│
├── 2. 事中控制（生成时）
│   ├── 强制引用：要求模型标注信息来源
│   ├── 推理链：让模型展示思考过程（CoT）
│   └── 多轮验证：复杂问题分步回答
│
└── 3. 事后检测（生成后）
    ├── 事实核查：与知识库对比
    ├── 自一致性：多次采样看答案是否一致
    └── 人工审核：关键场景人工确认
```

**详细回答：**

> "我构建了三层幻觉防御体系：
> 
> **事前防御**：首先用RAG提供准确的上下文，减少模型'编造'的可能；其次在Prompt里明确约束'如果信息不在上下文中，请回答无法确定'；最后降低temperature参数，让输出更确定。
> 
> **事中控制**：我要求模型在回答专业问题时使用推理链（Chain-of-Thought），展示思考过程，这样更容易发现逻辑漏洞。同时强制要求标注信息来源。
> 
> **事后检测**：我的RAG知识库里维护了'常见错误'列表，比如'AM CVn绝不含中子星'，会在后处理中检查模型输出是否包含这些错误。另外对关键数值（如周期、距离）会进行范围校验。
> 
> 实际效果上，这套机制将幻觉率从约15%降低到了5%以下。"

### 3.2 大模型基础问题

#### Q：Causal LM和Prefix LM有什么区别？

**回答要点：**

| 特性 | Causal LM | Prefix LM |
|------|-----------|-----------|
| 代表模型 | GPT系列 | T5、GLM |
| 注意力机制 | 严格自回归（只能看左边） | Prefix双向，生成部分自回归 |
| 训练目标 | 预测下一个token | Encoder-Decoder结构 |
| 适用场景 | 文本生成 | 翻译、摘要、理解任务 |

**详细回答：**

> "Causal LM（因果语言模型）和Prefix LM（前缀语言模型）是两种不同的语言模型范式。
> 
> **Causal LM** 如GPT系列，采用严格的自回归机制。在生成每个token时，只能看到它之前的token，使用掩码注意力实现。适合文本生成任务。
> 
> **Prefix LM** 如T5、GLM，对输入的Prefix部分使用双向注意力，对生成的部分使用自回归。它结合了Encoder的理解能力和Decoder的生成能力，适合翻译、摘要等任务。
> 
> 在我的项目中，对话Agent使用Causal LM进行生成，而RAG检索结果的编码可以用Prefix LM风格的Encoder来提升理解能力。"

#### Q：LoRA微调的原理是什么？

**回答要点：**

```
LoRA核心思想：W = W₀ + ΔW = W₀ + BA

其中：
- W₀：预训练权重（冻结，不训练）
- B, A：低秩矩阵（可训练，参数量很小）
- r：秩，通常8-64

优势：
1. 只训练BA，参数量减少到原来的 1/1000
2. 可以合并回原始模型，推理无额外开销
3. 不同任务训练不同LoRA，灵活切换
```

**详细回答：**

> "LoRA（Low-Rank Adaptation）的核心思想是：模型的权重更新ΔW可能是低秩的，可以用两个小矩阵的乘积BA来近似。
> 
> 具体来说，原始权重W₀保持不变，我们只训练低秩矩阵B和A。前向传播时：h = W₀x + BAx。
> 
> 这样做有三个好处：
> 1. **参数效率高**：如果原始矩阵是d×d，LoRA只需要训练2×d×r个参数，r通常只有8-64，参数量减少到原来的约1/1000。
> 2. **推理无开销**：训练完成后，可以把BA合并回W₀，推理速度和原来一样。
> 3. **模块化**：不同的任务训练不同的LoRA适配器，可以灵活切换。
> 
> 在我的模型蒸馏中，学生模型使用了LoRA进行微调，在单卡16GB显存上就能训练8B参数的模型。"

### 3.3 模型测试/评测问题（重点）

#### Q：大模型怎么评测？有哪些指标？

**回答框架：**

```
大模型评测体系
├── 1. 能力维度评测
│   ├── 知识理解：MMLU、C-Eval
│   ├── 推理能力：GSM8K（数学）、HumanEval（代码）
│   ├── 多语言能力：多语言翻译、理解任务
│   └── 安全性：TruthfulQA、安全性测试集
│
├── 2. 评测方法
│   ├──  perplexity（困惑度）：衡量模型对文本的预测能力
│   ├──  BLEU/ROUGE：基于n-gram匹配的文本相似度
│   ├──  BERTScore：基于语义嵌入的相似度
│   └──  LLM-as-a-judge：用强模型评判弱模型
│
└── 3. 实际应用场景评测
    ├──  RAG评测：检索准确率、回答忠实度
    ├──  对话评测：多轮连贯性、上下文理解
    └──  特定任务：根据业务设计测试集
```

**详细回答：**

> "大模型评测需要从多个维度进行：
> 
> **能力维度**，常用公开benchmark如MMLU（多学科知识）、GSM8K（数学推理）、HumanEval（代码生成）等。这些测试集可以横向对比不同模型的能力。
> 
> **评测指标**方面：
> - **BLEU/ROUGE**：基于n-gram匹配，适合机器翻译、摘要等任务
> - **BERTScore**：基于BERT嵌入的语义相似度，更能捕捉语义
> - **Perplexity**：衡量模型对测试集的预测能力，越低越好
> - **LLM-as-a-judge**：用GPT-4等强模型作为评判者，适合开放式生成任务
> 
> **业务场景评测**更重要。比如我的天文分析系统，我设计了专门的测试集：
> - 包含100个天文专业问答对
> - 包含20个幻觉诱导问题（检查模型是否犯错）
> - 包含10个多轮对话场景（检查上下文理解）
> 
> 评测时结合自动指标和人工评估，确保结果可靠。"

#### Q：LLM-as-a-judge有什么偏见问题？

**回答要点：**

```
LLM-as-a-judge的偏见
├── 1. 位置偏见（Position Bias）
│   └── 倾向于给放在前面的答案更高分
├── 2. 冗长偏见（Length Bias）
│   └── 倾向于给更长的答案更高分
├── 3. 自我提升偏见（Self-Enhancement）
│   └── GPT-4给GPT-4生成的答案更高分
├── 4. 格式偏见（Format Bias）
│   └── 对特定格式（如Markdown列表）有偏好
└── 5. 一致性不足
    └── 同样的问题多次询问，分数可能波动
```

**缓解方法：**
- 交换位置多次评测取平均
- 明确评分标准，限制输出长度
- 使用多个不同的评判模型投票
- 与人类评估对齐，校准评判标准

**详细回答：**

> "LLM-as-a-judge虽然方便，但存在几个已知的偏见问题：
> 
> **位置偏见**：当让模型比较两个答案时，它倾向于给放在前面的答案更高分。缓解方法是交换位置多次评测。
> 
> **冗长偏见**：模型倾向于给更长的答案更高分，即使简短答案更准确。可以在prompt里明确要求'不要以长度评判质量'。
> 
> **自我提升偏见**：GPT-4给GPT-4生成的答案打分更高。缓解方法是使用多个不同的评判模型投票，或者用不相关的模型作为评判者。
> 
> **一致性不足**：同样的问题多次询问，评判结果可能有波动。建议对每个样本多次采样取平均。
> 
> 在我的项目中，我使用GPT-4评判时，会交换位置评测两次，同时用多个指标综合评估，减少单一评判的偏见影响。"

#### 题目1：最长无重复子串

```python
def lengthOfLongestSubstring(s: str) -> int:
    """
    滑动窗口解法
    时间复杂度：O(n)
    空间复杂度：O(min(m, n))，m是字符集大小
    """
    char_set = set()
    left = 0
    max_len = 0
    
    for right in range(len(s)):
        # 如果字符已在窗口中，移动左边界
        while s[right] in char_set:
            char_set.remove(s[left])
            left += 1
        
        # 加入当前字符
        char_set.add(s[right])
        max_len = max(max_len, right - left + 1)
    
    return max_len
```

**面试要点：**
- 说明为什么用滑动窗口而非暴力解法
- 解释时间复杂度为什么是O(n)
- 可以延伸：如果需要返回子串本身怎么做？

#### 题目2：链表反转

```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def reverseList(head: ListNode) -> ListNode:
    """
    迭代解法
    时间复杂度：O(n)
    空间复杂度：O(1)
    """
    prev = None
    curr = head
    
    while curr:
        next_temp = curr.next  # 暂存下一个节点
        curr.next = prev       # 反转当前指针
        prev = curr            # prev前移
        curr = next_temp       # curr前移
    
    return prev
```

**面试要点：**
- 能画出指针变化的过程图
- 能写出递归版本（作为加分项）
- 能处理边界情况（空链表、单节点）

#### 题目3：快速排序

```python
def quickSort(nums: list, left: int, right: int):
    """
    快速排序
    平均时间复杂度：O(n log n)
    最坏时间复杂度：O(n²)
    空间复杂度：O(log n)
    """
    if left < right:
        pivot_idx = partition(nums, left, right)
        quickSort(nums, left, pivot_idx - 1)
        quickSort(nums, pivot_idx + 1, right)

def partition(nums: list, left: int, right: int) -> int:
    """分区函数"""
    pivot = nums[right]  # 选最后一个作为pivot
    i = left - 1
    
    for j in range(left, right):
        if nums[j] <= pivot:
            i += 1
            nums[i], nums[j] = nums[j], nums[i]
    
    nums[i + 1], nums[right] = nums[right], nums[i + 1]
    return i + 1
```

**面试要点：**
- 解释分区的过程
- 分析时间复杂度和空间复杂度
- 知道最坏情况（已排序数组）和优化方法（随机pivot、三数取中）
