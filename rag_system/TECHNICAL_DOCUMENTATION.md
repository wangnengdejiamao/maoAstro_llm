## 项目概述

### 一句话介绍

**AstroRAG** 是一个为天文领域设计的**双轨检索增强系统**，它结合了"语义理解"和"关键词匹配"两种检索方式，让大模型能够：
- ✅ 更准确地回答专业问题
- ✅ 自动标注信息来源（引用）
- ✅ 检测并减少"胡说八道"（幻觉）

### 解决的核心问题

| 问题 | 传统方案 | AstroRAG方案 |
|-----|---------|-------------|
| 大模型不懂专业术语 | 直接问模型 | **先检索相关知识库** |
| 模型"胡说八道" | 人工检查 | **自动检测幻觉** |
| 不知道信息来源 | 无法追溯 | **自动标注引用** |
| 查不到相关内容 | 单一检索 | **双轨并行检索** |

## 为什么要做双轨RAG？

### 什么是RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

简单说就是：**先查资料，再回答**

就像学生考试时：
- 闭卷考试 = 传统大模型（纯靠记忆，可能记错）
- 开卷考试 = RAG（可以查资料，答案更准确）

### 为什么需要"双轨"？

想象你在图书馆找书：

| 检索方式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| **向量检索**（语义理解） | 理解同义词、近义词 | 可能漏掉精确术语 | "找关于恒星演化的书" |
| **关键词检索**（倒排索引） | 精确匹配专业术语 | 不理解语义相似 | "找关于'赫罗图'的书" |
| **双轨融合** | 两者优点结合 | 实现复杂 | **专业领域问答** |

### 双轨如何工作？

```
用户问题: "赫罗图上白矮星的位置"
        ↓
┌─────────────────┐    ┌─────────────────┐
│   向量检索轨道    │    │   关键词检索轨道  │
│  (语义理解)      │    │  (精确匹配)      │
├─────────────────┤    ├─────────────────┤
│ • 理解"赫罗图"   │    │ • 匹配"赫罗图"   │
│ • 理解"白矮星"   │    │ • 匹配"白矮星"   │
│ • 找到语义相似   │    │ • 找到精确包含   │
│   的文档        │    │   这些词的文档   │
└────────┬────────┘    └────────┬────────┘
         │                       │
         └───────────┬───────────┘
                     ↓
            ┌─────────────────┐
            │    结果融合层    │
            │  (加权+去重)     │
            └────────┬────────┘
                     ↓
            最终检索结果（带分数）
```

### 三个核心模块

#### 1. 向量存储模块 (Vector Store)

**作用**: 理解语义，找到意思相近的内容

**通俗解释**: 就像谷歌翻译能理解"猫"和"猫咪"是同一个意思

**技术细节**:
- 使用 **BAAI/bge-large-zh-v1.5** 嵌入模型
- 将文本转换为 1024 维向量
- 使用 **ChromaDB** 存储和检索
- 相似度计算使用 **余弦相似度**

**工作流程**:
```
文本: "赫罗图显示恒星演化"
  ↓
嵌入模型
  ↓
向量: [0.12, -0.34, 0.89, ...] (1024个数字)
  ↓
存储到向量数据库
```

#### 2. 倒排索引模块 (Keyword Index)

**作用**: 精确匹配专业术语

**通俗解释**: 就像字典的索引，快速找到包含特定词的所有文档

**技术细节**:
- 使用 **jieba** 进行中文分词
- 构建 **倒排索引**: 词 → 文档ID列表
- 使用 **TF-IDF** 算法计算相关性
- 支持中英文混合检索

**工作流程**:
```
文档1: "赫罗图显示恒星演化"
文档2: "白矮星在赫罗图左下角"

倒排索引:
  "赫罗图" → [文档1, 文档2]
  "恒星"   → [文档1]
  "白矮星" → [文档2]

查询 "赫罗图":
  → 立即找到文档1和文档2
```

#### 3. 混合检索器 (Hybrid Retriever)

**作用**: 融合两种检索结果，发挥各自优势

**融合算法**:
```python
# 分数归一化
向量分数 = 原始向量分数 / 最大向量分数
关键词分数 = 原始关键词分数 / 最大关键词分数

# 加权融合
最终分数 = 向量分数 × 0.6 + 关键词分数 × 0.4

# 双命中奖励
如果 向量命中 且 关键词命中:
    最终分数 ×= 1.1  # 提升10%
```

## 核心技术点

### 1. 嵌入模型选择

**为什么选择 BAAI/bge-large-zh-v1.5？**

| 模型 | 优势 | 劣势 | 选择理由 |
|-----|------|------|---------|
| OpenAI Ada-002 | 效果好 | 需要API、贵 | 成本考虑 |
| text2vec | 中文好 | 领域通用 | 不够专业 |
| **bge-large-zh** | **中文强、开源、免费** | 模型较大 | **综合最优** |

**技术参数**:
- 向量维度: 1024
- 支持序列长度: 512 tokens
- 在大规模中文语料上训练
- 对长文本理解能力强

### 2. 中文分词优化

**问题**: 天文领域有很多专业术语，普通分词会切错

**例子**:
- 普通分词: "白矮星" → "白" + "矮星" ❌
- 正确分词: "白矮星" → "白矮星" ✅

**解决方案**:
```python
# 添加自定义词典
jieba.add_word('白矮星', freq=1000)
jieba.add_word('赫罗图', freq=1000);
jieba.add_word('灾变变星', freq=1000);
```

### 3. 检索性能优化

**向量检索优化**:
- 使用 **HNSW** 索引算法（近似最近邻）
- 检索时间: O(log n) 而非 O(n)
- 10万条文档检索 < 100ms

**关键词检索优化**:
- 倒排索引常驻内存
- 检索时间: O(1) 常数时间
- 支持布尔查询: "赫罗图 AND 白矮星"

## 去幻觉机制

### 什么是"幻觉"？

**定义**: 大模型编造不存在的信息，但说得有模有样

**例子**:
```
用户问: "开普勒-452b 的公转周期是多少？"

模型幻觉回答: "开普勒-452b 的公转周期是 365.25 天，
               和地球非常相似。"

真实情况: 开普勒-452b 的公转周期是 384.8 天
```

### 幻觉检测机制

#### 1. 指示词检测

**原理**: 模型"没底气"时会用模糊词汇

**检测列表**:
```python
HALLUCINATION_INDICATORS = [
    "我不确定", "可能", "也许", "我猜", "好像",
    "我不清楚", "没有相关信息", "无法确定",
    "i'm not sure", "maybe", "perhaps", "i guess",
]
```

**实现**:
```python
def check_indicators(answer: str) -> List[str]:
    found = []
    for indicator in INDICATORS:
        if indicator in answer.lower():
            found.append(indicator)
    return found
```

#### 2. 数值验证

**原理**: 验证回答中的数字是否来自检索文档

**例子**:
```
模型回答: "该恒星的温度是 5778K"

检索文档: 包含 "温度约为 5800K"

验证: 5778K 与 5800K 接近 → 可能正确 ✅

模型回答: "该恒星的温度是 10000K"

验证: 10000K 与 5800K 差距大 → 可能幻觉 ⚠️
```

**实现**:
```python
def verify_numbers(answer: str, documents: List[str]) -> List[str]:
    # 提取回答中的数字
    numbers_in_answer = extract_numbers(answer)
    
    # 合并所有文档文本
    doc_text = " ".join(documents)
    
    unsupported = []
    for num in numbers_in_answer:
        if num not in doc_text:
            unsupported.append(num)
    
    return unsupported
```

#### 3. 综合置信度计算

```python
confidence = 0.0

# 指示词惩罚
if indicators_found:
    confidence += 0.2 * len(indicators_found)

# 数值不一致惩罚
if unsupported_numbers:
    confidence += 0.3 * len(unsupported_numbers)

# 判断是否有幻觉
has_hallucination = confidence > 0.3
```

### 去幻觉的应用场景

| 场景 | 处理方式 |
|-----|---------|
| 训练阶段 | 过滤高幻觉风险的数据 |
| 推理阶段 | 标记低置信度回答，提醒用户核实 |
| 评估阶段 | 计算模型幻觉率，持续优化 |

## 引用溯源

### 为什么要引用？

**场景**: 用户使用AI回答写论文/报告

**问题**: 如果AI说错了，用户可能不知道

**解决方案**: 每条信息都标注来源

### 引用格式

```
回答正文...

📚 参考来源:
[1] Abrahams et al. 2022 - Informing the Cataclysmic Variable Sequence.pdf, 第10页 (相关度: 0.95)
[2] Abril et al. 2020 - Disentangling cataclysmic variables.pdf, 第23页 (相关度: 0.87)
```

### 引用生成流程

```
检索结果
  ↓
提取元数据
  ├── 文件名
  ├── 页码
  ├── 文档内容
  └── 相关度分数
  ↓
格式化引用
  ↓
附加到回答末尾
```

### 引用价值

1. **可追溯**: 用户可以查看原文验证
2. **可评估**: 通过相关度分数判断信息可靠性
3. **学术合规**: 满足学术写作引用要求

## 面试常见问题

### Q1: 什么是RAG？为什么需要RAG？

**简短回答（30秒）**:
> RAG是检索增强生成，核心思想是"先查资料再回答"。大模型有知识截止和幻觉问题，RAG通过检索外部知识库，让模型基于准确信息回答，提高准确性和可信度。

**详细回答**:
- 传统大模型的问题: 知识有截止日期、可能胡说（幻觉）、不懂专业领域
- RAG的解决方式: 检索相关知识 → 拼接上下文 → 让模型基于检索内容回答
- 本项目创新: 双轨检索（语义+关键词），更适合专业术语

### Q2: 双轨RAG和普通RAG有什么区别？

**简短回答**:
> 普通RAG只用向量检索，可能漏掉精确术语。双轨RAG同时使用向量检索（理解语义）和关键词检索（精确匹配），两者互补，特别适合专业领域。

**举例说明**:
- 向量检索: 能理解"白矮星"和"white dwarf"是同一个东西
- 关键词检索: 能精确找到包含"赫罗图"的文档
- 双轨融合: 既能理解同义词，又不漏专业术语

### Q3: 如何检测和减少幻觉？

**三个层面**:

1. **指示词检测**: 检测"我不确定""可能"等模糊词汇
2. **数值验证**: 检查回答中的数字是否在检索文档中
3. **置信度评分**: 综合计算回答的可信度，低于阈值时提示用户

**项目中的实现**:
- 训练阶段: 过滤高幻觉风险的数据
- 推理阶段: 标记低置信度回答
- 系统层面: 要求模型基于检索内容回答，不依赖记忆

### Q4: 向量检索和关键词检索的优缺点？

| 维度 | 向量检索 | 关键词检索 |
|-----|---------|-----------|
| 原理 | 语义相似度 | 精确匹配 |
| 优点 | 理解同义词、近义词 | 精确、快速 |
| 缺点 | 计算成本高 | 不理解语义 |
| 适用 | 概念性问题 | 术语查询 |
| 本项目权重 | 60% | 40% |

### Q5: 系统如何处理中文专业术语？

**分词优化**:
- 使用jieba分词，添加自定义词典
- 添加天文专业词汇: "白矮星"、"赫罗图"、"灾变变星"
- 保留中英混合术语: "SED"、"CV"、"QPO"

**索引优化**:
- 同时索引中文和英文内容
- 向量模型BGE对中文支持好
- 关键词索引支持中英文混合查询

### Q6: 系统的性能如何？

**数据规模**:
- 索引文档: 20,000+ 条问答对
- 向量维度: 1024
- 存储: 向量库约 500MB

**检索性能**:
- 向量检索: < 100ms
- 关键词检索: < 10ms
- 总响应时间: < 200ms

### Q7: 如果让你优化，你会怎么做？

**短期优化**:
1. 添加更多专业术语到自定义词典
2. 优化向量模型（尝试领域特定的嵌入模型）
3. 添加缓存机制，加速常见查询

**长期规划**:
1. 多模态RAG: 支持图片、表格检索
2. 动态权重: 根据查询类型自动调整两种检索的权重
3. 反馈学习: 根据用户点击优化排序
